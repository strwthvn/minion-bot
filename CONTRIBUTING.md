# Участие в разработке

Рады, что хочешь помочь! Этот гайд написан для тех, кто только начинает — не бойся задавать вопросы.

## Подготовка окружения

1. Установи [Node.js](https://nodejs.org/) версии 22 или выше
2. Установи [Git](https://git-scm.com/downloads)
3. Форкни репозиторий на GitHub (кнопка **Fork** в правом верхнем углу)
4. Склонируй свой форк:

```bash
git clone https://github.com/ТВОЙ_ЮЗЕРНЕЙМ/minion-bot.git
cd minion-bot
npm install
```

5. Создай файл `.env` по образцу `.env.example` и заполни его данными своего тестового бота
6. Зарегистрируй команды: `npm run deploy`
7. Запусти бота: `npm run dev`

## Как добавить новую команду

Каждая команда — отдельный файл в `src/commands/`. Бот подхватывает их автоматически.

### Шаг 1: Создай файл

Создай файл `src/commands/hello.js` (название файла = название команды).

### Шаг 2: Напиши команду

```js
const { SlashCommandBuilder } = require('discord.js');

module.exports = {
  // Определение команды — что увидит пользователь в Discord
  data: new SlashCommandBuilder()
    .setName('hello')
    .setDescription('Приветствует пользователя'),

  // Обработчик — что произойдёт при вызове команды
  async execute(interaction) {
    await interaction.reply(`Привет, ${interaction.user.displayName}!`);
  },
};
```

Каждый файл команды **обязан** экспортировать два поля:
- `data` — объект `SlashCommandBuilder` с названием и описанием
- `execute(interaction)` — асинхронная функция-обработчик

### Шаг 3: Зарегистрируй команду

```bash
npm run deploy
```

Это отправит обновлённый список команд в Discord. После этого новая команда появится в меню `/`.

### Шаг 4: Проверь

Запусти бота (`npm run dev`) и попробуй вызвать команду в Discord.

## Архитектура: куда класть код

```
commands/    — определение и обработка slash-команд
handlers/    — обработчики Discord-событий (реакции, войс и т.д.)
services/    — бизнес-логика (работа с БД, построение embed'ов)
database/    — подключение к базе данных и схема
utils/       — вспомогательные функции
```

**Правила:**

- **Одна команда = один файл** в `commands/`
- **Логику выноси в сервисы.** Если команда работает с базой данных или делает что-то сложное — создай сервис в `services/`. Команда должна только принимать ввод и вызывать сервисы.
- **Обработчики событий** (реакции, войс) — в `handlers/`. Подключаются в `src/index.js`.
- **Конфигурация** (ID каналов, константы) — в `src/config.js`, а не хардкодом в логике.

## Работа с Git

### Создай ветку

Не работай напрямую в `master`. Для каждой задачи создавай отдельную ветку:

```bash
git checkout -b feat/my-new-feature
```

Названия веток:
- `feat/что-добавляешь` — новая функциональность
- `fix/что-чинишь` — исправление бага
- `refactor/что-меняешь` — рефакторинг без изменения поведения

### Делай коммиты

```bash
git add src/commands/hello.js
git commit -m "feat: add /hello command"
```

Формат сообщений коммитов:
- `feat: ...` — новая функциональность
- `fix: ...` — исправление бага
- `refactor: ...` — рефакторинг
- `docs: ...` — документация
- `ci: ...` — CI/CD, Docker

### Отправь и создай Pull Request

```bash
git push origin feat/my-new-feature
```

Затем на GitHub нажми **Compare & pull request** и опиши, что ты сделал.

## Код-стайл

- Используй `const` по умолчанию, `let` — когда нужно переприсвоить. `var` не используем.
- Асинхронные операции — через `async/await`.
- Обработка ошибок — `try/catch` с информативными сообщениями пользователю (используй `interaction.reply({ content: '...', flags: 64 })` для сообщений, видимых только вызвавшему).
- Не хардкодь ID каналов и ролей — выноси в `src/config.js`.

## Процесс ревью

1. Создай Pull Request
2. Ревьюер проверит код и оставит комментарии
3. Если есть замечания — внеси правки и запуши в ту же ветку (PR обновится автоматически)
4. После одобрения PR вливается в `master`

Не стесняйся спрашивать, если что-то непонятно — все когда-то начинали.
